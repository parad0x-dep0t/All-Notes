# 13. Domain Persistence

## Objective

After obtaining Domain Admin (or equivalent privileges), identify persistence mechanisms that survive password changes, reboots, or user account modifications. Persistence should only be performed when explicitly permitted by the engagement scope.

> **Rule:** Persistence is usually the last phase of an engagement. Focus on stealth, reliability, and understanding how each mechanism works.

---

# 1. Verify Domain Administrator Access

Before attempting persistence, verify privileged access.

Check:

- Domain Admin
- Enterprise Admin
- SYSTEM on Domain Controller

---

# 2. Golden Ticket

Prerequisites:

- KRBTGT NTLM Hash
- Domain SID

Create a Golden Ticket.

Verify:

- Access to Domain Controller
- Access after password changes
- Ticket lifetime

---

# 3. Silver Ticket

Prerequisites:

- Service Account NTLM Hash
- Service SPN
- Domain SID

Create service-specific tickets.

Useful services:

- CIFS
- HOST
- HTTP
- WSMAN
- MSSQLSvc

---

# 4. Pass-the-Ticket (PTT)

Inject existing Kerberos tickets into memory.

Sources:

- Exported `.kirbi`
- Dumped tickets
- Existing logon sessions

Useful tools:

- Rubeus
- Mimikatz
- Impacket

---

# 5. Skeleton Key (Legacy)

Inject a master password into LSASS.

Notes:

- Requires Domain Controller access.
- Exists only until LSASS restarts.
- Mostly useful for understanding older attack techniques.

---

# 6. AdminSDHolder Abuse

Review:

- AdminSDHolder ACLs
- SDProp propagation

Useful for maintaining privileged access through ACL inheritance.

---

# 7. ACL Persistence

Look for opportunities to grant persistent rights.

Interesting permissions:

- GenericAll
- GenericWrite
- WriteDACL
- WriteOwner
- AllExtendedRights

Targets:

- Users
- Groups
- OUs
- Computers

---

# 8. Resource-Based Constrained Delegation (RBCD)

If permitted:

- Configure RBCD
- Add attacker-controlled computer
- Request service tickets on behalf of users

Common abuse:

- MachineAccountQuota
- GenericWrite
- GenericAll

---

# 9. Shadow Credentials

If `msDS-KeyCredentialLink` is writable:

- Add a KeyCredential
- Authenticate using PKINIT
- Obtain a TGT without changing the user's password

Useful tools:

- Whisker
- Certipy
- pyWhisker

---

# 10. Active Directory Certificate Services (AD CS)

If AD CS exists:

Check for:

- ESC1
- ESC2
- ESC3
- ESC4
- ESC6
- ESC8
- ESC13

Certificate-based authentication can provide long-term persistence.

Useful tools:

- Certipy
- Certify

---

# 11. SID History Abuse

Review:

- SID History
- Trust relationships

Used in:

- Cross-domain attacks
- Forest persistence

---

# 12. Group Membership Persistence

Check whether you can add accounts to privileged groups.

Examples:

- Domain Admins
- Enterprise Admins
- Account Operators
- Backup Operators
- DNSAdmins

Also review nested group memberships.

---

# 13. Computer Account Persistence

Review:

- MachineAccountQuota
- Existing computer accounts
- Delegation settings

Abuse machine accounts where appropriate.

---

# 14. GPO Persistence

Review writable Group Policy Objects.

Possible abuses:

- Scheduled Tasks
- Startup Scripts
- Logon Scripts
- Registry Preferences

Useful tools:

- SharpGPOAbuse
- PowerGPOAbuse

---

# 15. Review Existing Persistence

Enumerate existing persistence mechanisms.

Check:

- ACLs
- Delegation
- AD CS
- Scheduled Tasks
- GPOs
- Service Accounts
- Shadow Credentials

---

## Tips

- Golden Tickets require the KRBTGT hash.
- Silver Tickets require only the target service account hash.
- Shadow Credentials are often quieter than password resets.
- AD CS misconfigurations can provide durable persistence without modifying passwords.
- Writable GPOs can affect many systems simultaneously.
- Always document every persistence mechanism used during an engagement.
